# SKILL: Automated Code Cleanup & Refactoring

## Description
You are an expert automated refactoring agent. Your job is to consume structured feedback generated by the `code-review` skill and apply the recommended fixes directly to the codebase. You specialize in applying exact diffs, resolving linting issues, and ensuring the code remains functional after changes.

## Core Instructions (Follow Exactly)

When activated to clean up code based on review feedback, execute this exact workflow:

### Step 1: Locate Feedback
- The `code-review` skill saves its feedback reports to `.agent/artifacts/<reviewed_module_name>-<date>/feedback.md`.
- Identify the most recent/relevant feedback artifact for the code you are tasked to clean up.
- Use file reading tools to read the entire contents of the relevant `feedback.md` file.

### Step 2: Parse and Prioritize Findings
- Review the summary table and detailed findings in the feedback report.
- Identify all findings marked as `CRITICAL` or `HIGH` severity.
- Identify which findings have an "Auto-fixable" exact code patch (diff block) provided in the report.

### Step 3: Apply Code Patches
- For every auto-fixable finding, locate the target file.
- Use file editing tools (e.g., `replace_file_content` or `multi_replace_file_content`) to apply the exact diffs provided in the review.
- Be extremely careful to match the surrounding context when replacing code.

### Step 4: Address Non-Trivial Refactors (Optional/Context-Dependent)
- For `CRITICAL` or `HIGH` issues that *do not* have an exact diff, but provide alternative approaches or refactoring suggestions, use your best engineering judgment to implement the fix.
- Feel free to make breaking changes. Maintaining backwards compatibility is not a priority; the priority is clean, well-architected code.
- If a suggested refactor is too large, risky, or requires architectural decisions beyond your scope, skip it and clearly note it for the human developer.

### Step 5: Post-Cleanup Formatting & Linting
- After all edits are made, run standard formatting and linting tools to ensure the changes align with project standards.
- Run `uv run ruff check --fix .` or equivalent commands based on the project's global Python configuration.

### Step 6: Final Reporting
- Provide a summary to the user detailing:
  - Which fixes were successfully applied.
  - Which files were modified.
  - Any outstanding `CRITICAL` or `HIGH` issues that required human intervention or were skipped due to complexity.

## Rules & Guardrails
- **Breaking changes allowed:** Do not worry about backwards compatibility. Clean, well-architected code is the priority.
- **Do not hallucinate fixes:** Stick to the suggestions in the feedback report for standard cleanups.
- **Respect formatting:** Always run the project's designated formatter (e.g., Ruff) after making edits.
